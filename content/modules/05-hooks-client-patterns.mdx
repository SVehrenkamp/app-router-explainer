# Hooks & client patterns

Client components did not change — your hooks knowledge transfers wholesale. What changed is the *routing surface* they talk to, one sharp edge around `useSearchParams`, and where long-lived client state (providers, caches) should live in a server-first tree.

## next/router → next/navigation

The old `useRouter` was a kitchen sink. The new API splits it into focused hooks — and drops the parts that made `_app` re-render on every navigation:

| Pages Router (`next/router`) | App Router (`next/navigation`) |
| --- | --- |
| `router.push / replace / back` | `useRouter().push / replace / back` |
| `router.query` (params + search mixed) | `params` prop (route params) + `useSearchParams()` (query string) |
| `router.pathname` | `usePathname()` |
| `router.events` | gone — compose `usePathname` + `useSearchParams` in an effect |
| `router.isReady` dance | gone — hooks resolve with real values |

<CodeDiff before="pages/ — one hook, stringly mixed" after="app/ — focused hooks">

```tsx
import { useRouter } from 'next/router'

function SearchBox() {
  const router = useRouter()
  // params and query arrive mixed together,
  // and are empty until router.isReady.
  const q = (router.query.q as string) ?? ''
  const submit = (next: string) =>
    router.push(`/search?q=${encodeURIComponent(next)}`)
  …
}
```

```tsx
'use client'
import { useRouter, useSearchParams } from 'next/navigation'

function SearchBox() {
  const router = useRouter()
  const searchParams = useSearchParams()
  const q = searchParams.get('q') ?? ''
  const submit = (next: string) =>
    router.push(`/store/search?q=${encodeURIComponent(next)}`)
  …
}
```

## The useSearchParams bailout

`useSearchParams` has a rule with teeth: on a statically-rendered route, a client component reading search params **can't know them at build time**. Without a `<Suspense>` boundary isolating that component, Next deopts the *entire page* to client-side rendering. The fix is never "make the page dynamic" by reflex — it's a boundary around the one component that reads the params.

Our search page sidesteps this the server-first way: the **page** reads `searchParams` (which makes the route dynamic — correct for search) and the client `SearchBox` only *writes* the URL:

## Providers under a high boundary

The reflex from `_app` is "providers at the top" — and that instinct survives, with one correction: the provider is a *client wrapper*, not a *client boundary*. `app/providers.tsx` renders `{children}` straight through, so the tree inside stays server-rendered while any client island below can still `useQuery` or read context. High provider: fine. High boundary: the thing stage 1 of the journey exists to escape.

## Third-party client libraries

A library that calls hooks or touches `window` needs to live behind a `'use client'` file. If the package ships no directive itself, wrap it: one small file with `'use client'` that re-exports what you need. Your server components then pass data (or children) through the wrapper — same slot pattern as module 4.

## Case study: one ProductContext, both worlds

The hardest provider question in a real migration isn't *where* the provider goes — it's the deeply-integrated context that every component reads through selector hooks. `useProductDisplayPrice()`, `useProductDiscountPercent()`: dozens of call sites, derived values, no appetite for a big-bang rewrite. Server components can't call `useContext` — so is that context a dead end?

No. The refactor is a **split, not a rewrite**, and it works because those hooks were doing two jobs: *locating* the data (environment-specific) and *deriving* values from it (pure). Four small files:

1. **Selectors** move to a pure module — no React imports. This is the valuable logic, and it doesn't change.
2. **The client half** is your existing context file, minus the derivation — provider and hooks keep their exact signatures. **Every existing consumer compiles untouched.**
3. **The server half** uses `React.cache()` as a request-scoped slot — seeded once, read anywhere below with no prop threading. Same missing-provider error contract as the hook. `server-only` poisons it against bundle leaks.
4. **The channel** bridges them: one server component that fetches once, seeds the slot, and renders the unchanged client provider with the same bundle (one serialization across the boundary — the client never refetches).

<CodeDiff before="Before — hook-only, client-locked" after="After — same API, both worlds">

```tsx
// ProductContext.tsx — locate AND derive
// fused together, so everything that
// touches price must be client code.
const ProductContext = createContext(null)

export function useProductDisplayPrice() {
  const bundle = useContext(ProductContext)
  return formatPrice(bundle.pricing.priceCents)
}
```

```tsx
// selectors.ts — pure, shared verbatim
export const selectDisplayPrice = (b) =>
  formatPrice(b.pricing.priceCents)

// client.tsx — existing hook, now thin
export const useProductDisplayPrice = () =>
  selectDisplayPrice(useProductBundle())

// server.ts — the mirror, via cache() slot
export const productDisplayPrice = () =>
  selectDisplayPrice(readBundle())
```

</CodeDiff>

The migration is then incremental by construction: adopt the channel route by route (it renders the legacy provider, so nothing breaks), and move consumers server-side one call site at a time — `useProductDisplayPrice()` becomes `productDisplayPrice()`, drop the `'use client'`, done.
